# Release notes

Before creating a `vX.Y.Z` tag, add `release-notes/vX.Y.Z.md`. The file is the
user-facing GitHub Release body and must explain what changed and why it matters.

Use the sections that apply:

- Features
- Fixes
- Reliability
- Security
- Compatibility
- Upgrade notes and migrations
- Known limitations

Call out breaking changes, new requirements, operator actions, and migrations
explicitly. A commit list is useful supplemental traceability, but it is not a
substitute for substantive release notes.
